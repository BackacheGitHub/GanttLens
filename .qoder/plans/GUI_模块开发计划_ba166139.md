# GUI 模块开发计划

## 依赖关系总览

```
Task 1 (Model 扩展)
  └→ Task 2 (GanttLayoutEngine) ← 核心接缝，最先做
       └→ Task 3 (GanttCanvasView 渲染)
            └→ Task 4 (交互: 选中 + 悬停)
                 └→ Task 5 (属性面板 + 编辑)
                      └→ Task 6 (Undo/Redo + 保存)
                           └→ Task 7 (统计 Tab)
```

---

## Task 1: Model 层扩展

**目标**：补全 core 模块中 GUI 所需的字段

- Task model 增加 `color` 字段（支持 `is colored in Green` 语法映射到 GUI）
- TaskStatus 增加 `BLOCKED` 状态（支持阻塞可视化）
- 更新 parser 相关逻辑，确保新字段正确解析
- **验证**：`mvn test -pl core` 全部通过，新增字段有对应测试用例

---

## Task 2: GanttLayoutEngine（核心接缝）

**目标**：纯 Java 布局计算类，可独立于 JavaFX 运行和测试

输入：`List<Task>` + `LayoutConfig`（日期范围、缩放比例、行高、起始坐标）
输出：`List<TaskLayout>`（每个任务的 x, y, width, height, progressWidth）

子任务：
1. 定义 `TaskLayout` record（taskId, x, y, width, height, progressWidth, isCriticalPath）
2. 定义 `LayoutConfig` record（startDate, endDate, pixelsPerDay, rowHeight, startY, labelColumnWidth）
3. 实现日期→像素换算：`dateToX(LocalDate)` / `xToDate(double)`
4. 实现任务条布局计算：遍历任务列表，计算每个任务的坐标和尺寸
5. 实现命中检测：`hitTest(double mouseX, double mouseY) -> Optional<String>` 返回 taskId
6. 实现关键路径标记（从 ProjectStats.criticalPath 获取）

**验证**：编写 GanttLayoutEngineTest，覆盖以下场景：
- 0 任务返回空列表
- 单任务坐标计算正确
- 多任务行排列不重叠
- 日期↔像素双向换算一致
- 命中检测命中/未命中
- 200 任务性能 < 10ms

---

## Task 3: GanttCanvasView（Canvas 渲染）

**目标**：将 GanttLayoutEngine 的输出绘制到 JavaFX Canvas

子任务：
1. 创建 `GanttCanvasView extends Canvas`
2. 实现 `render(List<TaskLayout>, List<Task>)` 方法
3. 绘制时间轴（日期刻度线，工作日/周末底色）
4. 绘制任务条（按状态着色：灰色=未开始，蓝色=进行中，绿色=已完成）
5. 绘制进度填充（深色/浅色对比）
6. 绘制关键路径高亮（红色边框 2px）
7. 绘制依赖箭头（灰色细线 + 箭头）
8. 绘制"今天线"（红色竖虚线）
9. 实现水平和垂直滚动支持（Scroll 事件 + 平移）
10. 与 MainController 集成，替换 Gantt Chart Tab 的占位符

**验证**：

1. `GanttColorMapperTest`（纯 Java 单元测试）：
   - PENDING → 浅灰色、IN_PROGRESS → 蓝色、COMPLETED → 绿色、BLOCKED → 橙色
   - null/未知状态 → 默认灰色

2. `GanttCanvasViewTest`（TestFX 无头模式）：
   - 渲染后 Canvas 宽高 > 0 且像素缓冲非空
   - 200 个任务渲染总耗时 < 200ms（`System.nanoTime()` 测量）
   - 关键路径任务的 Canvas 区域包含红色像素（采样边框坐标）

3. `GanttCanvasScrollTest`（TestFX 无头模式）：
   - 模拟滚轮事件后 Canvas 内容偏移量发生变化

---

## Task 4: 交互层（选中 + 悬停）

**目标**：实现任务条的选择和悬停反馈

子任务：
1. 创建 `TaskSelectionModel`：管理选中状态（单选/多选/清空）
2. 创建 `GanttInteractionHandler`：处理鼠标事件
   - 点击：调用 `GanttLayoutEngine.hitTest()` 获取 taskId → 更新 SelectionModel
   - 悬停：调用 hitTest → 高亮边框 + 显示 tooltip（任务名、日期、负责人、进度）
3. 选中任务的视觉反馈（高亮边框 + 轻微放大效果）
4. 左侧任务名称列表与选中状态联动

**验证**：

1. `TaskSelectionModelTest`（纯 Java 单元测试）：
   - selectTask → getSelectedIds 包含该 taskId
   - selectTask 后再 selectTask（单选模式）→ 仅保留最新
   - deselectTask → getSelectedIds 不再包含该 taskId
   - clearSelection → getSelectedIds 为空
   - getSelectedTask 返回完整 Task 对象（非空）

2. `GanttInteractionHandlerTest`（纯 Java 单元测试，Mock LayoutEngine）：
   - hitTest 返回 taskId 时 → SelectionModel 选中对应任务
   - hitTest 返回 empty 时 → SelectionModel 保持不变
   - 悬停坐标命中时 → tooltip 内容包含任务名、日期、负责人、进度

3. `SelectionSyncTest`（TestFX 无头模式）：
   - 通过 InteractionHandler 选中任务 → 任务列表中对应行的样式类包含 "selected"

## Task 5: 属性面板 + 任务编辑

**目标**：选中任务后在底部展示可编辑的属性面板

子任务：
1. 创建属性面板（Scene Graph 组件）：
   - 名称：TextField
   - 开始/结束日期：DatePicker
   - 负责人：ComboBox
   - 进度：Slider + TextField（双向绑定）
   - 依赖：FlowPane 标签列表 + "Add" 按钮
2. 选中任务时自动填充面板数据
3. 编辑属性时实时更新内存模型 → Canvas 重绘
4. 状态栏显示 "未保存更改 *" 标记

**验证**：

1. `PropertyPanelTest`（TestFX 无头模式）：
   - 选中任务 → TextField 显示任务名、DatePicker 显示正确日期、Slider 值等于进度百分比
   - 未选中任务 → 所有字段为空/disabled

2. `PropertyEditingTest`（TestFX 无头模式）：
   - 修改 DatePicker → 内存模型 Task.startDate 同步更新
   - 修改 Slider → 内存模型 Task.progress 同步更新
   - 修改 ComboBox（负责人）→ 内存模型 Task.assignee 同步更新
   - 编辑后 statusLabel 文本包含 "*"（未保存标记）

3. `PropertyEditingRedrawTest`（TestFX 无头模式）：
   - 修改日期后 → Canvas 像素缓冲版本号变化（证明触发了重绘）

## Task 6: Undo/Redo + 保存

**目标**：支持撤销/重做和持久化

子任务：
1. 设计 Command 接口和具体命令类（EditDateCommand, EditProgressCommand 等）
2. 实现 CommandStack（undo/redo 栈）
3. 每次属性编辑生成 Command 并 push 到栈
4. Ctrl+Z 撤销 → 内存模型回退 → Canvas 重绘
5. Ctrl+Y 重做 → 内存模型前进 → Canvas 重绘
6. 实现保存逻辑：将内存 GanttSchedule 序列化回 .puml 文件
7. 菜单栏 Edit 菜单集成 Undo/Redo/Save

**验证**：

1. `CommandStackTest`（纯 Java 单元测试）：
   - push(cmd) → canUndo() = true, canRedo() = false
   - undo() → 模型回退到 push 前的状态
   - redo() → 模型恢复到 undo 前的状态
   - push 新命令 → redo 栈清空
   - 栈空时 undo/redo → 抛异常或静默忽略

2. `EditDateCommandTest`（纯 Java 单元测试）：
   - execute → Task.startDate 变为目标值
   - undo → Task.startDate 恢复为原值

3. `EditProgressCommandTest`（纯 Java 单元测试）：
   - execute → Task.progress 变为目标值
   - undo → Task.progress 恢复为原值

4. `SaveRoundtripTest`（纯 Java 单元测试）：
   - 构造 GanttSchedule → 序列化为 .puml 字符串 → 重新解析 → 断言 tasks 数量、名称、日期、负责人、依赖完全一致

## Task 7: 统计 Tab

**目标**：实现统计面板，展示 ProjectStats 数据

子任务：
1. 项目概览卡片（任务数、总人天、工期）
2. 关键路径展示（任务链 + 总天数）
3. 资源分布条形图（每人工作量）
4. 过载警告列表 + 平衡建议
5. 与 MainController 集成，数据随解析结果更新

**验证**：

1. `StatsDataBindingTest`（纯 Java 单元测试）：
   - 用已知 GanttSchedule + AnalysisEngine 构造 ProjectStats
   - 断言概览卡片数据：任务数 = tasks.size()、总人天 = totalManDays()、工期 = criticalPath.totalDurationDays()
   - 断言关键路径任务链顺序与 CriticalPathResult.criticalTaskIds() 一致
   - 断言过载警告数量 = overloads.size()

2. `StatsRefreshTest`（TestFX 无头模式）：
   - 初始解析后显示统计值 X → 修改源码重新解析 → 统计值变为 Y
   - 断言两次显示值不同且均与 AnalysisEngine 输出一致
